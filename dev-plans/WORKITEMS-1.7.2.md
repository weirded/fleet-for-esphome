### New in 1.7.2

- [x] **#240** *(1.7.2-dev.10)* — *ESPHome 2026.7.0 fails to compile — every job pinned to it dies at install with `ERROR: No matching distribution found for esphome==2026.7.0`.* Found on hass-4: the whole target list (`athom-plug-1`, `co2-sensor-1`, `bbq-gas-valve`, …) failed identically, all `ver=2026.7.0`, before any firmware built. **Root cause:** ESPHome 2026.7.0 raised its Python floor to `>=3.12` (`requires_python: <3.15,>=3.12.0`; 2026.6.5 was `>=3.11`). Our server + worker Docker images run Python **3.11.15**, so pip filters 2026.7.0 out of the index entirely (`Ignored the following versions that require a different python version: 2026.7.0 Requires-Python <3.15,>=3.12.0`) and the version list stops at 2026.6.5. Not our code, not the YAML — an upstream runtime-floor bump. Masked in CI because CI already ran Python 3.12 while the shipped images were still 3.11. **Fix:** bump every base image (`ha-addon/Dockerfile`, `Dockerfile.standalone`, `client/Dockerfile`, `build.yaml`, `tests/Dockerfile.integration`, plus `refresh-deps.sh` / `threaddump-addon.sh` sidecars) from `python:3.11-slim` to `python:3.13-slim` (digest re-pinned in lockstep), regenerate both `requirements.lock` files under 3.13 via `refresh-deps.sh` (PY-8/PY-9 — clean, no macOS leak; only a dropped transitive comment changed), set `mypy.ini` target to 3.13, and align CI `python-version` 3.12→3.13 so tests exercise the shipping runtime. `IMAGE_VERSION` 9→10 + `MIN_IMAGE_VERSION` 9→10 (PY-4 — new base Python is a rebuild-required image change; deployed remote workers must rebuild before taking jobs again). **Second blocker, found while verifying:** with the Python floor cleared, ESP32 targets still failed to *compile* on 2026.7.0. ESPHome 2026.7 replaced PlatformIO with a **native ESP-IDF toolchain** (`esphome/espidf/`) that downloads `openocd-esp32` and verifies it by executing it during framework install; `openocd` links `libusb-1.0.so.0`, absent from our slim image, so the verify exits 127, the installer deletes the tool, and the whole ESP-IDF install fails (`RuntimeError: ESP-IDF 5.5.4 framework installation failure`) — killing every ESP32 compile. Non-ESP32 platforms (LibreTiny bk72xx) were unaffected. **Fix:** add `libusb-1.0-0` to the apt layer of all three runtime Dockerfiles (server, client, standalone). Verified end-to-end: `esphome compile esp32-arduino.yaml` on 2026.7.0 in the rebuilt client image now reports `Successfully compiled program` and emits `firmware.factory.bin`/`firmware.ota.bin`. 3.13 chosen over 3.12 for maximum runway under ESPHome's `<3.15` ceiling.

- [x] **#241** *(1.7.2-dev.10)* — *`scanner._resolve_esphome_config` Stage-2 fallback throws `TypeError: do_packages_pass() got an unexpected keyword argument 'skip_update'` on ESPHome ≥2026.6.* Surfaced in hass-4 logs as a wall of `Could not resolve config for .archive/*.yaml: TypeError (do_packages_pass() got an unexpected keyword argument 'skip_update')`. ESPHome removed the `skip_update` kwarg in 2026.6 — the signature is now `(config, *, command_line_substitutions=None)`. The kwarg was still passed unconditionally in the substitution-only fallback path, so any config that fails Stage-1 full validation (packages with missing includes, archived files without `secrets.yaml`) then dies in Stage-2 too, killing the metadata-recovery safety net. Masked for live configs because Stage 1 succeeds for them; only `.archive/` entries exposed it. **Fix:** probe `inspect.signature(do_packages_pass)` for the `skip_update` param and call the matching form — deliberately *not* a `try/except TypeError` retry, because `do_packages_pass` clones git repos and a blind retry would re-run those side effects after partial work.

- [x] **#239** *(1.7.2-dev.8)* — *All compiles fail with `Bundle creation failed … FileNotFoundError: '/data/esphome-versions/<ver>/bin/python'` after the in-container local worker evicts the server's active ESPHome venv.* Found on hass-4: every queued job failed at bundle creation and the local worker spun on HTTP 500 from `jobs/next`. Root cause is a **round-2 recurrence of #119**. The server lazy-installs the selected ESPHome version (here 2026.5.1, matching the HA ESPHome Device Builder add-on) into `/data/esphome-versions/` and caches the path in `scanner._server_esphome_bin`; `create_bundle` shells into `<venv>/bin/python` to run ESPHome's `ConfigBundleCreator`. The embedded local worker **shares that same directory** (`ESPHOME_VERSIONS_DIR=/data/esphome-versions` in `main.py`'s spawn) but runs jobs on per-device *pinned* versions — it never runs a job on the server's selected version, so from the worker's `MAX_ESPHOME_VERSIONS=1` LRU and the `disk_quota` "collapse-venvs-to-1-MRU" sweep that venv looks idle and gets `shutil.rmtree`'d first. `_server_esphome_bin` then dangles → every subsequent bundle 500s until the add-on restarts. #119 fixed this for the **Clean Cache** path only (`client._clean_build_cache` preserves any `bin/esphome` dir); the **LRU + disk-quota** eviction paths were still unguarded — and with `MAX_ESPHOME_VERSIONS` hardcoded to 1 (DQ.8) the worker now collapses to a single venv aggressively, so the collision is near-guaranteed whenever any device pins a non-selected version. **Fix:** the server publishes its active version to a `.server-active-version` sentinel in the shared dir (`version_manager.write_server_active_version`, called from `scanner.ensure_esphome_installed` on success); both worker eviction engines now treat those versions as pinned — `VersionManager._evict_lru` skips them (with an added infinite-loop guard in the install eviction loop for the all-pinned case) and `client._run_disk_quota_sweep` unions them into the `disk_quota` `PinnedSet`. Remote workers (own dir, no sentinel) read an empty set — no behavior change. Belt-and-suspenders: `create_bundle` self-heals by reinstalling the selected version if the venv's interpreter is ever missing, instead of failing every compile. Regression nets: `tests/test_client.py` (eviction preserves the server-active version; no infinite loop when only pinned remain) + `tests/test_esphome_unbundle.py` (the sentinel is written on install). Worker change is `.py`-only (auto-updatable) — no `IMAGE_VERSION` bump.
