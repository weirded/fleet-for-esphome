### New in 1.7.1

- [x] **#231** *(1.7.1-dev.2)* — "Authentication Expired" repair sent users to the wrong place. The reauth flow, the initial config flow, the `token_required` error, the `config_entry_reauth` fix-flow description, and the underlying `ConfigEntryAuthFailed` exception message in `coordinator.py:103-106` all directed users to "Settings → Add-ons → ESPHome Fleet → Configuration" — the HA Supervisor's add-on Configuration tab. The Server token actually lives in the add-on's own UI at **Settings → Authentication → Server token** (`SettingsDrawer.tsx:193-199`). Reported by BradleyFord in #108. Fix: rewrote the four affected entries in `strings.json` and `translations/en.json`, plus the exception message, to reference the add-on's Settings drawer path verbatim ("Settings → Authentication → Server token") so the in-product copy matches the visible UI labels.

- [ ] **#232** — Recommended-installation README doesn't account for HA Container / HA Core users. Reported by digdilem in [#109](https://github.com/weirded/distributed-esphome/issues/109): the user clicked the "Add to my Home Assistant" badge from `README.md`, hit a dead-end because their HA install (Container) has no Add-on Store, and reasonably concluded "the install instructions are broken." Real fix is documentation, not code: the add-on path (the `[![Add repository to my Home Assistant]]` badge + Supervisor flow) only works on **HAOS** and **HA Supervised** installs. Users on **HA Container** (Docker) and **HA Core** (Python venv) need the standalone Docker path further down the README. Add a callout immediately under the "As a Home Assistant add-on (recommended)" header that names the supported install types and points incompatible users straight at the standalone section; also tighten the "As a standalone Docker container" lead-in so it positively names HA Container / HA Core / non-HA as the audience instead of just "outside Home Assistant. Most people don't need this." Mirror the same shape into `ha-addon/DOCS.md`.

- [ ] **#234** — `GET /api/v1/jobs/next` returns HTTP 500 repeatedly for some freshly-registered workers. Reported by moellere in [#112](https://github.com/weirded/distributed-esphome/issues/112): one Kubernetes worker pod gets repeated 500s while sibling pods on the same Deployment (same image, same env) respond healthy. Reporter's hypothesis: the scheduler eligibility loop in `ha-addon/server/api.py::get_next_job` compares against `perf_score` / benchmark fields that are still `None` for a brand-new worker — when a `None` slips into a numeric comparison the request crashes, the worker never gets benchmark data because it never claims a job, and the failing branch becomes permanent. Fix: wrap the per-worker eligibility check in a try/except that logs the traceback at WARNING with the worker's `client_id` and falls through to "no job for you right now" (HTTP 204) instead of bubbling 500. Add a regression test in `tests/test_api_jobs_next.py` that exercises a worker with empty `perf_score` / empty `system_info` and asserts 204, not 500. Pairs naturally with #235's scheduler fix.

- [ ] **#235** — Scheduler defers all remote workers indefinitely when the local-worker has a higher `perf_score` and only one slot. Reported by moellere in [#111](https://github.com/weirded/distributed-esphome/issues/111): with `local-worker` at 1 slot + 3 remote workers (6 slots total) + 6 jobs queued, only the local-worker grinds the queue serially while remote workers sit idle. Root cause traced to the deferral rule in `get_next_job` ("among workers with equal job counts, defer if one with higher effective score has free slots") — `other_free` is briefly true between every job because `local-worker.active` drops 1→0 the instant a job finishes; in that microsecond window every remote worker yields. Fix: cap the deferral so when `queue_size > local_capacity` (the higher-scored worker simply cannot consume the queue alone), remote workers claim the overflow rather than yield. Reporter's mental model: "deferral should never leave the queue blocked when there's a worker with free slots eligible to do the work." Reproducible in unit test: 1-slot fast worker + 1 multi-slot slow worker + N>1 jobs → all jobs should not serialize on the fast worker. Pairs with #234.

- [ ] **#236** — Worker firmware-variant upload races the job-success transition; server returns HTTP 409 `{"error": "job_not_working"}`. Reported in [#129](https://github.com/weirded/distributed-esphome/issues/129) (Unraid + docker-compose worker, ESPHome Fleet 1.7.0): OTA succeeds and the device is flashed correctly, but both the `factory` and `ota` variant uploads back to the server fail with 409 and the UI shows the job as failed even though the device is fine. Root cause: the worker calls `submit_result(SUCCESS)` *before* uploading the firmware variants; once the job leaves WORKING state the server's variant-upload endpoint refuses. Fix in `ha-addon/client/client.py`: re-order so variant uploads complete (or are explicitly skipped) **while the job is still WORKING**, then call `submit_result(SUCCESS)` last. Alternative server-side fallback: accept variant uploads on a job whose terminal state was set within the last ~30 seconds; pick whichever lands cleaner. Regression test: integration test that drives a happy-path compile-and-flash and asserts the variants land in firmware storage AND the UI sees SUCCESS without intermediate failed-flash noise.
