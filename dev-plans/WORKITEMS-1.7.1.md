- [x] **#234** *(1.7.1-dev.10)* — `GET /api/v1/jobs/next` returns HTTP 500 repeatedly for some freshly-registered workers. Reported by moellere in [#112](https://github.com/weirded/distributed-esphome/issues/112): one Kubernetes worker pod gets repeated 500s while sibling pods on the same Deployment (same image, same env) respond healthy. Reporter's hypothesis: the scheduler eligibility loop in `ha-addon/server/api.py::get_next_job` compares against `perf_score` / benchmark fields that are still `None` for a brand-new worker — when a `None` slips into a numeric comparison the request crashes, the worker never gets benchmark data because it never claims a job, and the failing branch becomes permanent. Fix: wrap the per-worker eligibility check in a try/except that logs the traceback at WARNING with the worker's `client_id` and falls through to "no job for you right now" (HTTP 204) instead of bubbling 500. Add a regression test in `tests/test_api_jobs_next.py` that exercises a worker with empty `perf_score` / empty `system_info` and asserts 204, not 500. Pairs naturally with #235's scheduler fix.

- [x] **#235** *(1.7.1-dev.10)* — Scheduler defers all remote workers indefinitely when the local-worker has a higher `perf_score` and only one slot. Reported by moellere in [#111](https://github.com/weirded/distributed-esphome/issues/111): with `local-worker` at 1 slot + 3 remote workers (6 slots total) + 6 jobs queued, only the local-worker grinds the queue serially while remote workers sit idle. Root cause traced to the deferral rule in `get_next_job` ("among workers with equal job counts, defer if one with higher effective score has free slots") — `other_free` is briefly true between every job because `local-worker.active` drops 1→0 the instant a job finishes; in that microsecond window every remote worker yields. Fix: cap the deferral so when `queue_size > local_capacity` (the higher-scored worker simply cannot consume the queue alone), remote workers claim the overflow rather than yield. Reporter's mental model: "deferral should never leave the queue blocked when there's a worker with free slots eligible to do the work." Reproducible in unit test: 1-slot fast worker + 1 multi-slot slow worker + N>1 jobs → all jobs should not serialize on the fast worker. Pairs with #234.

- [x] **#236** *(1.7.1-dev.10)* — Worker firmware-variant upload races the job-success transition; server returns HTTP 409 `{"error": "job_not_working"}`. **Fix shape:** server-side grace window (60s) on the variant-upload endpoint, not the WORKITEMS-prescribed worker re-order — verified the worker source on develop (commit 9eed30b, "fix: 1.6.1 #6 + #9") already calls `_archive_firmware_to_server(...)` BEFORE every `_submit_result(success)`. The reporter's race comes from the server's timeout-checker flipping the job mid-upload (slow Unraid + compose, ~minute-long upload of factory + ota), not worker-side ordering. Server now accepts late uploads from the still-assigned worker for `_FIRMWARE_UPLOAD_GRACE_SECONDS = 60` past `finished_at`; logs at INFO so the path is observable. Regression net in `tests/test_api_firmware_grace.py` covers the four cases (within grace + accepted, outside grace + rejected, wrong worker + rejected, working-state + still works). Reported in [#129](https://github.com/weirded/distributed-esphome/issues/129) (Unraid + docker-compose worker, ESPHome Fleet 1.7.0): OTA succeeds and the device is flashed correctly, but both the `factory` and `ota` variant uploads back to the server fail with 409 and the UI shows the job as failed even though the device is fine. Root cause: the worker calls `submit_result(SUCCESS)` *before* uploading the firmware variants; once the job leaves WORKING state the server's variant-upload endpoint refuses. Fix in `ha-addon/client/client.py`: re-order so variant uploads complete (or are explicitly skipped) **while the job is still WORKING**, then call `submit_result(SUCCESS)` last. Alternative server-side fallback: accept variant uploads on a job whose terminal state was set within the last ~30 seconds; pick whichever lands cleaner. Regression test: integration test that drives a happy-path compile-and-flash and asserts the variants land in firmware storage AND the UI sees SUCCESS without intermediate failed-flash noise.

Symptom: every `--from-ghcr` deploy spends an extra ~3–4 minutes on scp + tar-extract over slow links — and on the laptop's slow-link day (matrix run #2 on 2026-05-06) the upload took 27 min before SSH timed out, killing the deploy entirely. Fix landed: changed every `--exclude='distributed-esphome/<x>'` to `--exclude='ha-addon/<x>'` so the patterns match the source path tar actually sees, and added the `__pycache__` / `.pytest_cache` / `.mypy_cache` / `.ruff_cache` excludes that the working `scripts/haos/install-addon.sh` sibling already had. Verified locally: 128 MB / 40 885 entries → **52 KB / 18 entries** (CHANGELOG, DOCS, icons, config, apparmor.txt, rootfs/, translations/, build.yaml, VERSION — exactly what Supervisor reads). Side-bonus: dev-checkout `server/` and `client/` `.py` files no longer leak into `/usr/share/hassio/addons/local/distributed-esphome/` on hass-4 in image-based mode. **Fix-shipping note:** `push-to-hass-4.sh` is `.gitignore`d (line 72: "Local deployment scripts" — its sibling `push-to-haos.sh` is tracked). The fix lives only on this laptop and won't propagate via develop.
